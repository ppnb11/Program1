🔎 漏洞明细与修复方案

#漏洞名称等级状态修复措施
1SQL注入 — 搜索接口 🔴 高危 ✅ 已修复 搜索接口由 f-string 拼接 SQL 改为参数化查询（? 占位符），用户输入无法改变 SQL 语句结构。UNION 注入、盲注均被拦截。
2SQL注入 — 注册接口 🔴 高危 ✅ 已修复 注册接口由 f-string 拼接 SQL 改为参数化查询，用户输入不再拼入 SQL 语句。
3数据库明文密码存储 🔴 高危 ✅ 已修复 注册和初始化时使用 generate_password_hash() 对密码进行 bcrypt/scrypt 哈希存储，数据库泄露也无法还原原密码。
4登录/注册数据不同步 🔴 高危 ✅ 已修复 登录验证从硬编码 USERS 字典改为统一查询 SQLite 数据库，注册的用户现在可以正常登录。
5UNION注入提取密码 🔴 高危 ✅ 已修复 参数化查询 + 搜索改为仅选取 id, username, email, phone 字段，不包含 password 字段，即使注入成功也无法获取密码。
6注册无频率限制 🟡 中危 ✅ 已修复 注册接口添加 @limiter.limit("10 per minute") 频率限制，第 11 次请求返回 HTTP 429。
7无密码强度校验 🟡 中危 ✅ 已修复 添加 validate_password() 函数，密码长度至少 6 位，未达标时返回友好提示。
8无输入校验 🟡 中危 ✅ 已修复 添加 validate_username()、validate_email()、validate_phone() 函数，用户名长度 3-20 位，限制特殊字符，校验邮箱格式。
9错误信息泄露数据库结构 🟡 中危 ✅ 已修复 捕获 SQL 异常，返回通用错误信息 "注册失败，用户名可能已存在"，不暴露 UNIQUE constraint 等数据库细节。
10搜索返回含密码字段 🟡 中危 ✅ 已修复 搜索结果由 SELECT * 改为只选取 id, username, email, phone，密码字段不再进入模板数据。
