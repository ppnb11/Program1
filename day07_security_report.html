�️ CSRF 防护机制

本系统通过以下多层机制实现对 CSRF（跨站请求伪造）攻击的全面防御：

#防护机制技术实现原理说明
1 全局 CSRFProtect 中间件 csrf = CSRFProtect(app) 在 Flask 应用初始化时全局注册 拦截所有 POST/PUT/PATCH/DELETE 请求，自动验证 CSRF Token 是否存在且有效。无效请求直接返回 400。
2 表单内嵌 CSRF Token 所有敏感表单中添加 <input type="hidden" name="csrf_token" value="{{ csrf_token() }}"> 每次渲染页面时生成唯一的随机 Token，嵌入表单隐藏字段。提交时与 session 中存储的 Token 比对。
3 Token 绑定 Session Flask-WTF 将 CSRF Token 存储在服务端 session 中（session['csrf_token']） 攻击者跨域无法读取或预测 Token（同源策略限制 + 随机性保证）。即使诱导用户点击恶意链接，也无法伪造有效请求。
4 Session Cookie 安全属性 Flask session cookie 受 httpOnly + SameSite=Lax 保护 JS 无法读取 session cookie（httpOnly），跨站请求不携带 cookie（SameSite），双重阻断 CSRF 攻击路径。
5 自定义请求头隐式 Token WTF_CSRF_CHECK_DEFAULT = True，自动从 X-CSRFToken 请求头提取 Token 支持 AJAX 请求场景。自定义请求头受同源策略保护，跨域请求无法携带，进一步阻断 CSRF。
6 Referer 校验 Flask-WTF 默认启用 Referer 头同源检查（可配置 WTF_CSRF_SSL_STRICT） 验证请求来源 Referer 与当前 host 一致。跨域 CSRF 攻击的 Referer 不匹配，请求被拒绝。

CSRF 攻击流程 vs 防御拦截

步骤攻击者操作防御拦截点结果
①构造恶意网页，内含自动提交表单指向 /change-password——
②诱导已登录用户访问恶意页面——
③恶意表单自动提交 POST 请求SameSite=Lax → 跨站请求不携带 session cookie❌ 无身份
④请求到达服务器CSRFProtect 中间件 → 检查请求中是否有 csrf_token 字段❌ Token缺失
⑤伪造请求被处理Referer 校验 → 请求来源与服务器 host 不一致❌ 来源非法
✅ 结论：在当前配置下，攻击者在第③步即被 SameSite Cookie 策略拦截，即使绕过该层，第④步 CSRFProtect 中间件和第⑤步 Referer 校验也会逐层拦截。三层纵深防御确保 CSRF 攻击不可行。

🔒 额外安全措施建议：

1. 验证密码复杂度：增加大写字母、数字、特殊字符的混合要求
2. 密码修改后失效所有 session：修改密码后强制所有设备重新登录，防止 session 劫持
3. 密码修改频率限制：添加 flask_limiter 限流，防止暴力修改
4. Referer/Origin 头校验：作为纵深防御，额外验证请求来源
5. 密码修改审计日志：记录每次密码修改的时间、IP、User-Agent
6. 敏感操作二次认证：修改密码等敏感操作要求重新输入密码后再验证
